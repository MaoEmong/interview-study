01자바
›JVM, JRE, JDK는 뭐가 다른가요?자바가 플랫폼 독립적인 이유까지

JVM은 자바 바이트코드를 각 운영체제에 맞게 실행해주는 가상 머신이고, JRE는 여기에 실행에 필요한 라이브러리를 더한 실행 환경입니다. JDK는 JRE에 컴파일러 같은 개발 도구까지 포함한 개발 키트입니다. 자바 코드는 바이트코드로 컴파일된 뒤 각 OS의 JVM이 해석해서 실행하기 때문에, 한 번 작성하면 플랫폼에 상관없이 동작합니다.

›== 과 equals() 의 차이는?그리고 hashCode() 를 같이 오버라이드해야 하는 이유

== 는 두 변수가 같은 객체를 참조하는지 주소를 비교하고, equals() 는 값이 동등한지를 비교합니다. 그리고 equals()를 재정의할 때는 hashCode()도 함께 재정의해야 하는데, HashMap이나 HashSet이 hashCode() 로 먼저 버킷을 찾고 그 안에서 equals()로 비교하기 때문입니다. 둘을 맞추지 않으면 논리적으로 같은 객체인데도 컬렉션에서 다른 객체로 취급되는 문제가 생깁니다.

›가비지 컬렉션은 어떻게 동작하나요?

가비지 컬렉션은 더 이상 참조되지 않는 객체를 JVM이 자동으로 찾아 메모리를 회수하는 기능입니다. GC Root에서 참조를 따라가며 도달할 수 있는 객체는 남기고, 닿지 않는 객체를 수거하는 방식입니다. 또 대부분의 객체는 금방 사라진다는 점을 이용해서 Young 영역과 Old 영역으로 나눠 효율적으로 관리합니다.

›오버로딩과 오버라이딩의 차이는?

오버로딩은 같은 이름의 메서드를 매개변수의 개수나 타입을 다르게 해서 여러 개 정의하는 것이고, 오버라이딩은 상속받은 부모의 메서드를 자식 클래스에서 같은 형태로 재정의하는 것입니다. 오버로딩은 컴파일 시점에 호출될 메서드가 정해지고, 오버라이딩은 런타임에 실제 객체 타입에 따라 정해진다는 차이도 있습니다.

›제네릭을 왜 쓰나요? 컴파일 타임에 어떤 이점이 있죠?

제네릭은 타입을 미리 지정해서 컴파일 시점에 타입 안정성을 보장하기 위해 사용합니다. 잘못된 타입이 들어가면 런타임이 아니라 컴파일 단계에서 바로 잡을 수 있고, 데이터를 꺼낼 때 형변환을 하지 않아도 돼서 코드가 더 안전하고 간결해집니다.

02네트워크
›TCP와 UDP의 차이는?TCP가 신뢰성을 보장하는 방법 (3-way handshake)

TCP는 연결을 먼저 맺고 데이터의 순서 보장과 재전송을 통해 신뢰성을 보장하는 프로토콜이고, UDP는 연결 없이 빠르게 데이터를 보내는 데 집중한 프로토콜입니다. TCP는 통신 전에 3-way handshake로 연결을 확인하기 때문에 파일 전송이나 웹처럼 정확성이 중요한 곳에 쓰이고, UDP는 실시간 스트리밍이나 게임처럼 약간의 손실이 있더라도 속도가 중요한 곳에 적합합니다.

›주소창에 google.com 치고 엔터를 누르면 무슨 일이 벌어지나요?

먼저 DNS 조회를 통해 도메인 주소를 IP 주소로 변환합니다. 그다음 그 IP로 TCP 연결을 맺고, HTTPS라면 추가로 암호화 과정을 거칩니다. 연결이 되면 브라우저가 HTTP 요청을 보내고 서버가 HTML 같은 응답을 돌려줍니다. 마지막으로 브라우저가 받은 HTML과 CSS, JS를 해석해서 화면에 렌더링합니다.

›HTTP와 HTTPS의 차이는?

HTTPS는 기존 HTTP에 암호화를 더한 것입니다. 통신 내용을 암호화하기 때문에 중간에서 데이터를 가로채거나 변조하는 것을 막을 수 있습니다. 그래서 비밀번호나 결제 정보처럼 민감한 데이터를 안전하게 주고받을 수 있고, 인증서를 통해 접속한 서버가 신뢰할 수 있는 곳인지도 확인합니다.

›GET과 POST의 차이는?멱등성 관점까지

GET은 주로 데이터를 조회할 때 쓰고 파라미터가 URL에 노출되며, POST는 데이터를 생성하거나 변경할 때 쓰고 데이터를 본문에 담아 보냅니다. GET은 여러 번 요청해도 결과가 같은 멱등성을 가지지만, POST는 요청할 때마다 리소스가 생성될 수 있어 멱등하지 않습니다. 그래서 민감한 정보나 변경 작업은 POST로 처리하는 것이 맞습니다.

›쿠키, 세션, 토큰(JWT)은 각각 어떻게 다른가요?

쿠키는 브라우저에 저장되어 요청마다 서버로 함께 전송되는 작은 데이터입니다. 세션은 로그인 상태 같은 정보를 서버가 직접 저장하고, 클라이언트는 세션 ID만 쿠키에 들고 다니는 방식입니다. 반면 JWT는 사용자 정보를 토큰 자체에 담아 클라이언트가 보관하고 서버는 서명만 검증하기 때문에, 서버가 상태를 따로 저장하지 않아도 돼서 확장에 유리합니다.

03MySQL · 데이터베이스
›인덱스는 왜 빠른가요? 단점은? 어떤 자료구조를 쓰죠?B-Tree

인덱스는 책의 색인처럼 전체 데이터를 다 훑지 않고 원하는 데이터를 빠르게 찾게 해주는 기능이고, 보통 B-Tree 구조를 사용해 정렬된 상태로 탐색합니다. 다만 조회는 빨라지지만 데이터를 추가하거나 수정할 때마다 인덱스도 갱신해야 해서 쓰기 성능이 떨어지고 저장 공간도 더 쓰게 됩니다. 그래서 조회가 자주 일어나는 컬럼에 선별적으로 거는 것이 중요합니다.

›트랜잭션과 ACID에 대해 설명해보세요.

트랜잭션은 더 이상 나눌 수 없는 하나의 작업 단위로, 계좌 이체에서 출금과 입금이 한 묶음이 되는 것이 예시입니다. 이를 안전하게 보장하는 성질이 ACID인데, 원자성은 전부 성공하거나 전부 실패하는 것, 일관성은 작업 후에도 데이터 규칙이 지켜지는 것, 격리성은 동시에 실행돼도 서로 간섭하지 않는 것, 지속성은 커밋된 데이터가 장애가 나도 유지되는 것을 의미합니다.

›정규화를 왜 하나요? 반정규화는 언제 하죠?

정규화는 데이터 중복을 없애 무결성을 지키기 위해 테이블을 나누는 과정입니다. 중복을 줄이면 데이터를 수정할 때 생기는 이상 현상을 막을 수 있습니다. 다만 테이블이 나뉘면서 JOIN이 많아져 조회 성능이 떨어질 수 있는데, 이럴 때 성능이 더 중요하면 일부러 중복을 허용하는 반정규화를 적용하기도 합니다.

›JOIN의 종류와 동작 방식은?

JOIN은 두 테이블을 특정 컬럼을 기준으로 연결해 하나의 결과로 합치는 것입니다. INNER JOIN은 양쪽 모두에 매칭되는 행만 가져오고, LEFT JOIN은 왼쪽 테이블은 전부 가져오면서 오른쪽은 매칭되는 것만, 없으면 NULL로 채웁니다. RIGHT JOIN은 그 반대로 동작합니다.

04스프링 부트
›IoC와 DI가 뭐고, 왜 좋은가요?

IoC는 객체의 생성과 관리를 개발자가 아니라 스프링 컨테이너가 담당하도록 제어권을 넘기는 개념이고, DI는 필요한 객체를 직접 생성하지 않고 외부에서 주입받는 방식입니다. 이렇게 하면 객체 간 결합도가 낮아져서 구현체를 갈아 끼우기 쉽고, 테스트할 때 가짜 객체를 주입하기도 편해집니다.

›Bean의 생명주기는? 왜 싱글톤으로 관리되나요?

Bean은 컨테이너가 생성하고 의존성을 주입한 뒤 초기화, 사용, 소멸 순서로 관리합니다. 기본적으로 싱글톤으로 관리되는데, 매 요청마다 객체를 새로 만들지 않고 하나를 공유함으로써 메모리와 성능 면에서 이점이 있기 때문입니다. 대신 여러 곳에서 공유되기 때문에 Bean은 상태를 갖지 않도록 설계하는 것이 안전합니다.

›AOP는 어떤 문제를 풀려는 건가요?

AOP는 로깅, 트랜잭션, 보안처럼 여러 곳에서 반복되는 공통 기능을 핵심 비즈니스 로직과 분리하기 위한 방식입니다. 같은 코드를 모든 메서드마다 넣는 대신 한 곳에 정의해두고 필요한 지점에 적용하는 방식이라, 코드 중복을 줄이고 유지보수가 쉬워집니다. 대표적으로 @Transactional 이 AOP를 활용한 예시입니다.

›@Transactional 은 내부적으로 어떻게 동작하나요?프록시

@Transactional 은 스프링이 해당 객체를 감싸는 프록시를 만들어서, 메서드 호출 전후로 트랜잭션을 시작하고 커밋하거나 롤백하는 방식으로 동작합니다. 메서드가 정상적으로 끝나면 커밋하고, 예외가 발생하면 롤백합니다. 다만 프록시 기반이기 때문에 같은 클래스 안에서 메서드를 직접 호출하면 적용되지 않는 점은 주의해야 합니다.

›디스패처 서블릿을 중심으로 요청 처리 흐름을 설명해보세요.

스프링 MVC는 모든 요청을 DispatcherServlet이 먼저 받는 프론트 컨트롤러 구조입니다. 요청이 들어오면 HandlerMapping으로 처리할 컨트롤러를 찾고, 컨트롤러가 로직을 실행한 뒤 결과를 반환하면 ViewResolver를 통해 화면을 그리거나 JSON으로 응답합니다.

05프론트 · HTML/CSS/JS
›SSR과 CSR은 뭐가 다른가요?서버사이드 렌더링 vs 클라이언트사이드 렌더링

SSR은 서버가 완성된 HTML을 만들어 보내는 방식이고, CSR은 빈 페이지와 JS를 받아 브라우저가 화면을 그리는 방식입니다. SSR은 첫 화면이 빠르게 보이고 검색 엔진 최적화에 유리하며, CSR은 첫 로딩은 다소 느리지만 이후 페이지 전환이 부드럽고 서버 부담이 적다는 장점이 있습니다.

›AJAX가 뭔가요? 어떤 문제를 풀려고 등장했죠?

AJAX는 페이지 전체를 새로고침하지 않고 필요한 데이터만 서버와 비동기로 주고받아 화면 일부만 갱신하는 기술입니다. 예전에는 작은 변경에도 페이지 전체를 다시 받아야 했는데, AJAX 덕분에 더 부드러운 사용자 경험이 가능해졌습니다. 검색어 자동완성이나 무한 스크롤이 대표적인 예시입니다.

›DOM이 뭐고, JS로 DOM을 어떻게 조작하나요?

DOM은 HTML 문서를 객체 트리 형태로 표현한 것으로, JS가 이 트리에 접근해서 요소를 추가하거나 수정, 삭제할 수 있습니다. 보통 querySelector 로 원하는 요소를 찾고 textContent 나 classList 같은 속성으로 조작합니다. 다만 DOM 조작은 비용이 크기 때문에 잦은 변경은 최소화하는 것이 좋습니다.

›동기와 비동기의 차이는? 콜백·프로미스·async/await 는 어떻게 발전해왔나요?

동기는 한 작업이 끝나야 다음으로 넘어가는 방식이고, 비동기는 작업이 끝나기를 기다리지 않고 다음을 진행하다가 완료되면 결과를 처리하는 방식입니다. 자바스크립트에서는 처음에 콜백으로 비동기를 처리했는데 중첩되면 복잡해지는 콜백 지옥 문제가 있었고, 이후 프로미스로 체인 형태로 개선됐으며, async/await가 등장하면서 비동기 코드를 동기 코드처럼 읽기 쉽게 작성할 수 있게 됐습니다.

›fetch 로 REST API를 호출하는 흐름을 설명해보세요.

fetch 로 URL에 요청을 보내면 Promise가 반환되고, 응답을 json() 메서드로 파싱한 뒤 async/await 나 then 으로 결과를 처리합니다. GET이 아닌 POST 요청이나 헤더, 본문은 fetch의 두 번째 인자에 옵션으로 전달합니다. 그리고 네트워크 오류나 응답 상태 코드를 res.ok 로 확인해서 예외 처리도 함께 해주는 것이 좋습니다.

06도커 · 쿠버네티스
›컨테이너와 가상머신은 뭐가 다른가요?

가상 머신은 운영체제를 통째로 가상화하기 때문에 무겁고 실행이 느린 반면, 컨테이너는 호스트의 OS 커널을 공유하면서 애플리케이션과 필요한 환경만 격리하기 때문에 가볍고 빠릅니다. 그래서 컨테이너는 부팅이 몇 초 안에 끝나고 자원을 적게 써서, 같은 서버에 더 많은 컨테이너를 띄울 수 있습니다.

›이미지와 컨테이너의 관계는?

이미지는 실행에 필요한 것들을 담은 정적인 템플릿이고, 컨테이너는 그 이미지를 실제로 실행한 인스턴스입니다. 객체지향에 비유하면 클래스와 객체의 관계와 비슷하고, 하나의 이미지로 여러 개의 컨테이너를 띄울 수 있습니다.

›쿠버네티스는 어떤 문제를 풀어주나요?오케스트레이션이 왜 필요한가

컨테이너가 많아지면 배포하고 확장하고 장애를 복구하는 일을 사람이 일일이 관리하기 어려워지는데, 쿠버네티스는 이런 작업을 자동화해주는 오케스트레이션 도구입니다. 자동 배포와 롤백, 트래픽에 따른 자동 확장, 그리고 컨테이너가 죽으면 다시 띄워주는 기능 등을 제공합니다.

›Pod가 뭔가요?

Pod는 쿠버네티스에서 배포하는 가장 작은 단위로, 하나 이상의 컨테이너를 묶은 그룹입니다. 같은 Pod 안의 컨테이너들은 네트워크와 스토리지를 공유하며, 쿠버네티스는 컨테이너가 아니라 이 Pod 단위로 배포하고 관리합니다.

07MSA
›모놀리식과 MSA의 장단점은?

모놀리식은 모든 기능이 하나의 큰 애플리케이션으로 묶인 구조이고, MSA는 기능별로 잘게 나눈 독립적인 서비스들의 모음입니다. 모놀리식은 개발과 배포가 단순하지만 규모가 커지면 작은 변경에도 전체를 다시 배포해야 하는 부담이 있고, MSA는 서비스별로 독립적인 배포와 확장이 가능하지만 서비스 간 통신과 운영 복잡도가 올라간다는 단점이 있습니다.

›서비스 간 통신 방식은?동기 REST vs 비동기 메시지

크게 동기 방식과 비동기 방식이 있습니다. 동기 방식은 REST나 gRPC로 요청하고 응답을 기다리는 방식이라 단순하지만, 상대 서비스에 장애가 생기면 영향을 받습니다. 비동기 방식은 메시지 큐에 메시지를 넣고 각자 처리하는 방식이라 결합도가 낮고 장애에 강하지만, 전체 흐름을 추적하기 어렵다는 특징이 있습니다.

›분산 환경에서 트랜잭션을 어떻게 다루나요?Saga 패턴

MSA에서는 서비스마다 데이터베이스가 분리되어 있어 하나의 트랜잭션으로 묶기 어렵습니다. 그래서 Saga 패턴을 사용하는데, 각 서비스가 자신의 작업을 순차적으로 처리하고 중간에 실패하면 앞서 완료한 작업들을 되돌리는 보상 트랜잭션을 실행해서 데이터 정합성을 맞춥니다.

›API 게이트웨이의 역할은?

API 게이트웨이는 클라이언트의 모든 요청을 받는 단일 진입점으로, 요청을 알맞은 서비스로 라우팅해주는 역할을 합니다. 또 인증이나 로깅, 요청 제한 같은 공통적인 처리를 한곳에서 담당하기 때문에, 각 서비스는 자신의 핵심 로직에만 집중할 수 있습니다.

08플러터
›플러터의 렌더링 방식은 네이티브와 어떻게 다른가요?자체 렌더링 엔진

플러터는 운영체제가 제공하는 네이티브 UI 컴포넌트를 사용하지 않고, Skia나 Impeller 같은 자체 렌더링 엔진으로 화면을 직접 그립니다. 그래서 iOS와 안드로이드에서 동일한 화면을 보장할 수 있고, 하나의 코드로 양쪽 플랫폼을 모두 커버할 수 있습니다.

›StatelessWidget과 StatefulWidget의 차이는?

StatelessWidget은 상태가 없어서 한 번 그려지면 변하지 않는 위젯이고, StatefulWidget은 내부 상태가 바뀌면 화면을 다시 그리는 위젯입니다. 예를 들어 버튼을 누를 때마다 숫자가 바뀌는 카운터는 Stateful로 만들고, 고정된 텍스트나 아이콘은 Stateless로 만듭니다.

›위젯 트리란? "모든 게 위젯"이라는 말의 의미는?

플러터에서는 버튼이나 텍스트뿐 아니라 정렬, 여백, 패딩 같은 레이아웃 요소까지 전부 위젯입니다. 이 위젯들이 부모와 자식 관계로 중첩되면서 위젯 트리를 이루고, 화면은 이 위젯들을 조립해서 구성됩니다. 상태가 바뀌면 트리를 다시 빌드해서 화면을 갱신합니다.

›상태관리를 왜 따로 하나요?

앱이 커지면 여러 화면이 같은 데이터를 공유해야 하는데, StatefulWidget만으로는 상태를 위젯 트리 위아래로 계속 넘겨야 해서 복잡해집니다. 그래서 Provider나 Riverpod, Bloc 같은 상태관리 도구로 상태를 한곳에서 관리하고, 그 상태가 필요한 위젯만 구독해서 갱신하도록 합니다. 이렇게 UI와 상태를 분리하다 보면 자연스럽게 일정한 아키텍처 패턴으로 정리하게 됩니다.

›플러터에서 MVVM 패턴을 어떻게 적용하나요?

MVVM은 화면을 담당하는 View, 화면에 보여줄 상태와 로직을 담당하는 ViewModel, 데이터를 담당하는 Model로 역할을 분리하는 패턴입니다. 플러터에서는 위젯이 View가 되고, ViewModel이 상태와 비즈니스 로직을 가지며, View는 ViewModel을 구독해서 상태가 바뀌면 화면을 갱신합니다. 이렇게 UI와 로직을 분리하면 코드 역할이 명확해지고 테스트와 유지보수가 쉬워지며, Provider나 Riverpod 같은 상태관리 도구와 함께 자주 사용합니다.

09AI 활용
›개발할 때 AI를 어디에, 어떻게 활용했나요?구체적인 사례 중심으로

저는 단순히 코드를 받아 쓰는 것을 넘어서 컴파운드 엔지니어링 관점으로 접근했습니다. 한 번의 작업으로 끝내지 않고, 반복되는 작업은 skill로 정의하고 AI가 일관되게 동작하도록 rule을 작성해두며, 도구를 호출하면서 작업을 수행하는 하네스 환경을 구성했습니다. 이렇게 한 번의 작업이 다음 작업의 자산으로 쌓이도록 만들어서, 쓸수록 효율이 누적되는 방식으로 활용했습니다.

›AI가 준 코드를 그대로 쓰지 않고 검증한 방법은?

저는 AI가 준 코드를 그대로 신뢰하지 않고, 먼저 직접 읽으면서 동작 원리를 이해하려고 합니다. 그리고 테스트를 통해 실제로 의도대로 동작하는지 확인하고, 필요하면 공식 문서와 대조합니다. 스스로 왜 이렇게 동작하는지 설명할 수 없는 코드는 사용하지 않는다는 기준을 가지고 있습니다.

›환각(할루시네이션)을 어떻게 걸러내나요?

AI가 실제로 존재하지 않는 API나 잘못된 문법처럼 그럴듯하지만 틀린 정보를 줄 수 있다는 점을 항상 염두에 둡니다. 그래서 공식 문서나 실제 실행을 통해 교차 검증을 하고, 출처가 불분명한 답변은 의심합니다. 또 작은 단위로 나눠서 검증하며 진행하면 잘못된 정보가 누적되는 것을 막을 수 있습니다.
